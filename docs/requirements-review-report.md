## 🔴 Blocker

### B1. FR-01 与 AC-01 / Section 8 存在致命矛盾：登录方式到底是"验证码"还是"密码"？

**问题描述**：
- FR-01 写的是"提供手机号 + **验证码**登录接口，验证通过后返回 JWT token"
- AC-01 写的是"传入正确**凭证**"（暗示密码），AC-02 写的是"传入错误的手机号或**密码**"
- Section 8 "不做的事"明确写了"图形验证码 / 短信验证码...放 Phase 2 配合微信手机号授权"
- staff 表有 `password_hash` 字段（bcrypt 密码哈希），没有验证码相关字段

**为什么这是 Blocker**：认证是系统的入口。如果评审通过后开发人员按 FR-01 实现验证码登录，但数据库设计按密码登录，会导致返工。且验收标准无法执行（到底按哪种方式验收？）。这是整个需求的根基性矛盾。

**修改建议**：将 FR-01 修正为"提供手机号 + 密码登录接口，验证通过后返回 JWT token"，与 AC-01、staff 表设计、Section 8 对齐。

---

### B2. 3D 模型 URL 为空时的降级策略未定义

**问题描述**：
- 车型表 `car_model` 中 `3d_model_url` 字段为 `NULL`（允许为空）
- AC-09 只说车型列表返回含 `3d_model_url` 字段，未定义该字段为 NULL 时的行为
- FR-12 将 `3d_model_url` 列为车型字段，但未定义缺失时的降级策略
- Section 9 "风险与开放项" 提到"3D 模型文件的上传和管理 - 待确认，初期可由平台管理员手动上传到 OSS 后填写 URL"——恰恰因为初期模型覆盖不完整，NULL 是大概率事件
- 没有任何验收标准覆盖"3d_model_url 为 NULL 时，API 应返回什么？客户端应展示什么？"

**为什么这是 Blocker**：3D 看车是 WrapLab 的核心差异化功能。如果车型没有 3D 模型，门店销售打开小程序看到什么——白屏？报错？占位图？这直接决定了 MVP 是否可用。按角色定义 Checklist，3D 模型加载失败的降级策略是 Blocker 级检查项。

**修改建议**：
- 新增 FR：定义当 `3d_model_url` 为 NULL 时，API 返回 `null`，客户端展示"3D 模型暂未上线"占位提示（或回退到车型图片）
- 新增 AC：`Given 车型的 3d_model_url 为 NULL，When 查询该车型详情，Then 返回数据中 3d_model_url = null，客户端展示模型缺失占位提示而非报错`
- 在 Section 8 "不做的事" 中补充说明：Phase 1 不保证所有车型都有 3D 模型

## 🟡 Should Fix

### S1. 验收标准缺少对超长输入、特殊字符、重复提交的边界覆盖

**问题描述**：
- AC 覆盖了空数据 (AC-10/11)、参数缺失 (AC-17)、参数校验 (AC-53)、分页边界 (AC-54)，但没有覆盖：
  - **超长输入**：`customer_name` 传 1000 个字符怎么办？`note` 传 10MB 文本怎么办？
  - **特殊字符/注入攻击**：`customer_name` 含 `<script>alert(1)</script>` 或 SQL 注入 payload 怎么办？
  - **重复提交/快速双击**：同一方案被连续 POST 两次怎么办？会创建两条记录吗？

**为什么这很重要**：缺少边界验收标准 = 开发人员自行决定边界行为 = 安全隐患和数据质量风险。NFR-14 提到了"防止 SQL 注入和 XSS"，但没有对应的 AC 来验证。

**修改建议**：
- 新增 AC：`Given customer_name 长度超过 100 字符，When 创建改色方案，Then 返回 400 "客户姓名不能超过 100 字符"`
- 新增 AC：`Given note 包含 <script> 标签，When 创建改色方案，Then 方案创建成功但 script 标签被转义或过滤（XSS 防护）`
- 新增 AC：`Given 同一店员短时间内连续 POST 两次相同参数的创建方案请求，When 两次请求到达，Then 两次均成功创建（不报错），或后端实现幂等性去重`——至少需要明确行为

---

### S2. P0/P1 优先级定义缺失，导致 FR-14 存在歧义

**问题描述**：
- 文档使用 P0 和 P1 标记每个 FR 的优先级，但从未定义 P0/P1 的含义
- FR-14 "车型数据查询接口无需鉴权（公开读取）" 标记为 **P1**，但在 API 接口速览 (Section 7) 中，`GET /api/v1/vehicles/brands` 等接口列为共享接口且无鉴权标注
- 如果 FR-14 是 P1（非 P0），是不是意味着 Phase 1 MVP 里车型接口反而**需要**鉴权？这与 Section 7 的设计矛盾

**为什么这很重要**：优先级定义缺失会造成开发排期混乱——开发团队不知道该先做哪个。FR-14 的 P1 标记与实际 API 设计不一致，可能导致车型接口被错误地加上鉴权。

**修改建议**：
- 在文档开头定义优先级：`P0 = MVP 必须实现，缺失则业务不可用 | P1 = MVP 应该实现，可延后但建议包含 | P2 = Phase 2`
- 将 FR-14 修正为 P0（车型数据公开读取是 MVP 基础能力），或如果确实是 P1，则调整 Section 7 的 API 设计使其与优先级一致

---

### S3. 非功能需求缺少 API 限流（Rate Limiting）和数据备份策略

**问题描述**：
- NFR 覆盖了安全（NFR-10~14）、错误处理（NFR-30~36）、性能（NFR-40~43），但缺少：
  - **API 限流**：登录接口无频率限制，可被暴力破解；创建方案接口无频率限制，可被恶意刷数据
  - **数据备份/恢复**：没有定义数据库备份策略。如果数据库损坏，业务数据如何恢复？

**为什么这很重要**：限流是安全基线——没有限流的登录接口等同于给暴力破解开绿灯。数据备份是可靠性基线——MVP 上线后如果发生数据丢失且无备份，业务损失不可逆。

**修改建议**：
- 新增 NFR：`登录接口限流：同一 IP 每分钟最多 10 次请求，超出返回 429 Too Many Requests`
- 新增 NFR：`业务写接口（配置/报价）限流：每用户每分钟最多 30 次请求`
- 新增 NFR：`数据库每日自动备份，保留最近 7 天备份，备份文件存储于独立于数据库服务器的存储（如 OSS）`

### S4. 色卡颜色列表接口 (FR-21) 缺少分页，高颜色数场景存在性能风险

**问题描述**：
- FR-21 定义"根据色卡品牌 ID 获取该品牌下的颜色列表"，无分页参数
- API 接口 `GET /api/v1/colors/swatches?brandId=` 无 `page`/`size` 参数
- 对比：改色方案列表 (FR-33) 和报价单列表 (FR-44) 都设计了分页
- 现实中一个色卡品牌（如 AX）可能有 200+ 种颜色

**为什么这很重要**：200 个颜色加上每个颜色 8 个字段（含 rgb 分量、单价等），单次请求的数据量可能达到几十 KB。在高频查询场景（门店销售频繁切换品牌浏览颜色），没有分页会导致不必要的带宽消耗和客户端渲染压力。需要决策：是不分页（一次加载完，客户端本地过滤），还是加分页？

**修改建议**：
- 明确决策：颜色列表是否需要分页？如果不需要，在需求中说明理由（如"客户端需本地搜索/过滤颜色，故全量返回"）
- 如果需要分页（如颜色 > 50），则补充分页参数并新增对应 AC

## 💭 Nice to Have

### N1. Admin 角色的 store_id 处理逻辑未显式声明

**问题描述**：JWT 中包含 `store_id`（FR-02），但 admin 角色跨门店操作。admin 登录时 `store_id` 是什么值？NULL？0？特殊标记？这在 FR-02 中没有说明，也未在 user story 中体现 admin 登录场景。

**建议**：在 FR-02 或 NFR-02 中补充：`admin 角色的 JWT 中 store_id 为 0 或 NULL，全局拦截器不对 admin 角色的查询进行 store_id 过滤`。否则开发人员会困惑：admin 的 store_id 该填什么？

---

### N2. staff 表 phone 字段 UNIQUE 约束的隐含影响

**问题描述**：`staff` 表中 `phone` 字段标记为 `UNIQUE`。这意味着同一个手机号不能同时在门店 A 和门店 B 注册。这是一个合理的全局唯一约束，但隐含一个业务场景：一个销售人员同时在两家门店兼职怎么办？Phase 1 可能不需要支持，但值得在"不做的事"或"风险"中标注。

**建议**：在 Section 8 "不做的事"中补充：`店员跨门店兼职（同一手机号多门店）`——理由：Phase 1 假设店员只属于一个门店。

---

### N3. 全车标准面积 15m2 的假设过于简化

**问题描述**：FR-41 写"全车标准面积暂定为 15 平方米"，Section 9 也标注了此风险。但 15m2 对于紧凑型轿车（约 12m2）和大型 SUV（约 22m2）差异很大。如果 MVP 只用 15m2 报价，SUV 车主的报价会严重偏低。

**建议**：考虑在 Phase 1 中至少按 `body_type`（sedan=15, suv=20, hatchback=12）做一个粗粒度的面积映射，而非一刀切 15m2。这样可以避免 SUV 报价偏差过大。

---

### N4. OSS 文件上传后的访问鉴权策略未定义

**问题描述**：FR-60 提供 OSS 上传，FR-61 返回 URL。但没有说明返回的 URL 是公开 URL 还是签名 URL（有过期时间）。如果是签名 URL，客户端缓存 URL 后过期会导致 3D 模型加载失败（又回到 B2 的问题）。

**建议**：在 FR-61 中补充：`3D 模型文件 OSS URL 为公开读权限（或长期有效签名 URL，有效期 >= 7 天），确保客户端缓存后仍可访问`。

---

### N5. 数据库列名 `3d_model_url` 以数字开头

**问题描述**：`car_model` 表定义了字段 `3d_model_url`。MySQL 中，以数字开头的列名必须用反引号包裹（`` `3d_model_url` ``），否则会报语法错误。虽然这是实现细节，但如果架构设计阶段漏掉这一点，建表 SQL 会出错。

**建议**：考虑将字段名改为 `model_3d_url` 或 `model_url_3d`，避免数字开头。或者在文档中增加一条 NFR 提醒"DDL 中数字开头的列名需用反引号包裹"。
